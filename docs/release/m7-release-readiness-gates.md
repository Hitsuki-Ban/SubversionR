# M7 Release Readiness Gates

M7 release readiness gates define when SubversionR may claim public product readiness. The `0.2.4` Windows Beta package, live digest attestation, and Marketplace pre-release publication are complete, while the `0.2.5` candidate remains pending release, live attestation, and Marketplace publication; this document does not claim that public installation, signing, signed source-to-binary provenance, final SBOM/NOTICE/legal review, migration, rollback, or full TortoiseSVN integration is complete.

The current intermediate target is narrower: Windows `win32-x64` Beta packaging readiness for local file-backed SVN workflows. That target keeps public release blockers intact while requiring installed VSIX evidence for the implemented Beta workflows, a minimum state-engine performance baseline, and current claim/evidence alignment.

## Release Channels

- Stable: supported public release channel. It requires every blocking gate in this document to be closed.
- Pre-release: public candidate channel for completed product builds that still need platform and native dependency validation.
- Internal/CI: private validation channel with additional assertions and trace output. It is not a public demo channel.

No channel may publish an intentionally incomplete demo build as if it were product-ready.

The Windows Beta is available through the public Marketplace pre-release channel and as a GitHub Releases VSIX. It is not a Stable channel and keeps signing/provenance, previous-stable rollback, cross-platform support, broad remote/auth matrices, coverage-guided fuzzing, merge, merge preview, and mergeinfo as explicit non-claims.

## Blocking Gates

Public release is blocked until:

- All P0/P1 requirements have explicit evidence or an approved release exception. This includes `PRD-010`, `PRD-012`, and `PRD-015`.
- Structured requirement status has no unexplained unknowns. The requirements catalog recording specification approval lives in the private planning archive since the public cutover; `docs/release/requirements-release-evidence.csv` records release evidence, exceptions, and blockers for every P0/P1 requirement. The public smoke gate verifies the evidence CSV's own integrity, and the maintainer-side `scripts/release/verify-requirement-catalog-alignment.ps1` gate cross-checks it against the archived catalog before release approval.
- Automatic PR checks run the lightweight TypeScript, Rust, localization, support, readiness smoke, script, and M7l8 no-build/no-run native remote fuzz target preflight gates in `.github/workflows/pr-fast.yml`. The heavy release workflow in `.github/workflows/ci.yml` stays manual until native build, packaging, live vulnerability review, fixed-seed fuzz build/run, signing, and Marketplace work are split into product-ready gates.
- Differential, integration, VS Code E2E, performance, security, migration, and release-install gates are complete.
- No high-severity data corruption, secret leakage, Extension Host hang, or unbounded backend failure remains open.
- Documentation and UI behavior are consistent in English, Japanese, and Chinese.
- Protocol v1.29 operation failures preserve a bounded symbolic libsvn error chain through the required native ABI, render localized actionable notifications without raw internal codes, expose a bounded redacted `SubversionR` log with Show Log, and keep failed commit message/review selection state. Missing commit messages use one explicit localized prompt; cancellation has no operation side effects, and epoch/generation/path/changelist state is revalidated before RPC dispatch.
- Security issue handling is published in `SECURITY.md`.

## Platform Packaging Gates

Platform packaging is blocked until:

- Each platform-specific VSIX contains exactly the sidecar, C bridge, libsvn, and native dependencies for that target.
- The `win32-x64` staged VS Code package layout gate passes with relative-path artifact hashes, exact backend sidecar/bridge paths, an allowlisted private libsvn/APR/OpenSSL/iconv runtime set, AMD64 PE machine checks, a fail-closed packaged-native startup probe that enforces the extension's current minimum protocol and capability contract, loads the packaged bridge, and completes a read-only libsvn-backed discovery operation, a validated 128x128-or-larger PNG Marketplace icon, SVN-focused Marketplace keywords, and no bundled SVN CLI fixture tools anywhere in the staged package. Stage, layout verification, and VSIX packaging reject stale protocol daemons and bridges missing required C ABI symbols; there is no probe skip switch or compatibility fallback. The root `package.json` version is authoritative: the extension and Cargo workspace declarations must be strict SemVer and exactly equal, and packaged/installed evidence must report the matching daemon and `subversionr-svn-bridge/<version>` labels while preserving libsvn's independent version. Missing, malformed, prerelease-drifted, or unequal declarations fail before packaging without rewriting or inferring a version.

- Core native workflows do not depend on system `svn`, TortoiseSVN, PATH probing, registry probing, or random system libsvn.
