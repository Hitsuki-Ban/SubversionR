# M7 Release Readiness Gates

M7 release readiness gates define when SubversionR may claim public product readiness. This document does not claim that packaging, signing, SBOM, migration, rollback, or full TortoiseSVN integration is complete.

The current intermediate target is narrower: Windows `win32-x64` Beta packaging readiness for local file-backed SVN workflows. That target keeps public release blockers intact while requiring installed VSIX evidence for the implemented Beta workflows, a minimum state-engine performance baseline, and current claim/evidence alignment.

## Release Channels

- Stable: supported public release channel. It requires every blocking gate in this document to be closed.
- Pre-release: public candidate channel for completed product builds that still need platform and native dependency validation.
- Internal/CI: private validation channel with additional assertions and trace output. It is not a public demo channel.

No channel may publish an intentionally incomplete demo build as if it were product-ready.

The internal Windows Beta package is not a Stable or Pre-release public channel. It may be used only after the local workflow installed E2E and performance gates are recorded, and it must keep Marketplace/public install, signing/provenance, previous-stable rollback, cross-platform support, broad remote/auth matrices, coverage-guided fuzzing, merge, merge preview, and mergeinfo as explicit non-claims.

## Blocking Gates

Public release is blocked until:

- All P0/P1 requirements have explicit evidence or an approved release exception. This includes `PRD-010`, `PRD-012`, and `PRD-015`.
- Structured requirement status has no unexplained unknowns. `Reference/requirements.csv` records specification approval, and `docs/release/requirements-release-evidence.csv` records release evidence, exceptions, and blockers for every P0/P1 requirement.
- Automatic PR checks run the lightweight TypeScript, Rust, localization, support, readiness smoke, script, and M7l8 no-build/no-run native remote fuzz target preflight gates in `.github/workflows/pr-fast.yml`. The heavy release workflow in `.github/workflows/ci.yml` stays manual until native build, packaging, live vulnerability review, fixed-seed fuzz build/run, signing, and Marketplace work are split into product-ready gates.
- Differential, integration, VS Code E2E, performance, security, migration, and release-install gates are complete.
- No high-severity data corruption, secret leakage, Extension Host hang, or unbounded backend failure remains open.
- Documentation and UI behavior are consistent in English, Japanese, and Chinese.
- Security issue handling is published in `SECURITY.md`.

## Migration And Rollback Gates

Migration and rollback release blockers:

- Versioned cache schema with an explicit delete-and-reconcile rollback policy that never mutates a working copy during rollback (`MIG-008`).
- The M7d foundation exposes protocol cache schema `subversionr.cache.v1`, deletes only extension-owned cache roots, and stores a user-visible cache migration report. It is not sufficient by itself to claim full release upgrade/rollback completion.
- The M7f fixture hashes a fixture-local `.svn/wc.db` sentinel before and after package install, upgrade, and rollback to keep working-copy non-mutation observable in CI. This is not a substitute for previous-stable installed-product migration E2E.
- Upgrade and rollback tests from the previous stable release once a stable release exists (`MIG-010`).
- User-visible migration report for imported or changed settings and command behavior (`MIG-011`).
- Legacy compatibility only where `Reference/legacy_migration.csv` explicitly requires it.
- No silent semantic change from legacy commands or settings.
